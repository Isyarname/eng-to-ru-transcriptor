Этот документ описывает внутреннее устройство библиотеки для разработчиков и контрибьюторов.

## 🔄 Пайплайн обработки

### 1. Инициализация `Transcriber`

При создании объекта `Transcriber()` ресурсы **не загружаются**. Только запоминаются пути:
- `rules_path` — путь к правилам транслитерации (по умолчанию — ресурс пакета)
- `dict_path` — путь к словарю исключений (по умолчанию — ресурс пакета)
- `cache_dir` — путь к кэшу (по умолчанию — `user_cache_dir`)
- `custom_exceptions` — пользовательские исключения (если переданы)

### 2. Первый вызов `transcribe()`

- **Компиляция правил.** Если `compiled_rules.json` в кэше отсутствует, правила транслитерации из `transliteration_rules.py` парсятся, макросы разворачиваются, дублируются варианты для заглавных букв, результат сохраняется в JSON-кэш.
- **Загрузка словаря.** Читается `en_dict.txt`, парсится в `dict[str, str]`, сохраняется в `_dict_cache`.
- **Объединение словарей.** Встроенный + пользовательские исключения (пользовательские имеют приоритет).

### 3. Последующие вызовы

- Скомпилированные правила и словарь берутся из памяти (`_compiled_rules`, `_dict_cache`).
- `gruut` импортируется лениво — только если есть слова, которых нет ни в одном словаре.

### Где что лежит

| Что                | Где                                  | Почему                              |
|--------------------|--------------------------------------|-------------------------------------|
| Исходный код       | `src/eng_to_ru_transcriber/`         | src-layout — стандарт PyPA          |
| Словари и правила  | `src/eng_to_ru_transcriber/data/`    | Попадают в wheel при сборке         |
| Кэш правил         | `user_cache_dir/`                    | read-write, не часть пакета         |
| Пример текста      | `examples/text.txt`                  | Демонстрация, не ресурс пакета      |

## 🏗️ Архитектурные решения

### Почему класс `Transcriber`, а не функция?

- **Явное состояние.** Понятно, что загружено в память, что можно перезагрузить.
- **Управление жизненным циклом.** `reload_dictionary()`, `reload_transliteration()`, `clear_cache()`.
- **Тестируемость.** Можно инжектить свои пути и словари через `__init__`.
- **Несколько экземпляров.** Каждый со своими пользовательскими исключениями.

### Почему пользовательские исключения дополняют встроенный словарь?

Встроенный `en_dict.txt` содержит тысячи слов. Если бы пользовательские исключения заменяли его, пришлось бы вручную переносить все нужные слова. Дополнение позволяет точечно переопределять произношение конкретных слов (имена, бренды, термины), сохраняя стандартную функциональность.

### Три уровня исключений

1. **Встроенный словарь** (`en_dict.txt`) — база, всегда загружается.
2. **Объект-уровневые** (`custom_exceptions` в `__init__`, `add_exception()`) — дополняют встроенный, имеют приоритет при конфликте ключей.
3. **Временные** (`custom_exceptions` в `transcribe()`) — используются только для одного вызова, не сохраняются в объекте.

Каждый следующий уровень перезаписывает предыдущий при совпадении ключей.

### Почему src-layout?

Защищает от случайного импорта локальной папки вместо установленной версии. Принцип "test what you ship".

### Почему кэш в `user_cache_dir`, а не в пакете?

После `pip install` пакет лежит в `site-packages/`, куда у пользователя нет прав на запись. Кэш должен быть в пользовательской директории.

### Почему два типа нормализаций IPA?

- **Подстрочные** (`eɪ → ej`) — применяются внутри слов.
- **Словарные** (`tə → tu`) — только для целых слов, чтобы не сломать `təmɔɹoʊ`.

### Почему `gruut` импортируется лениво?

`gruut` подгружает ML-модель (~50-100 МБ). Если все слова есть во встроенном словаре, `gruut` вообще не загружается, и импорт пакета занимает миллисекунды.

### Почему `transliteration_rules.py` — это `.py`, а не `.txt`?

Правила транслитерации удобно описывать как Python-переменные (`macros`, `rules`). Файл загружается через `importlib.util.spec_from_file_location` — как данные, а не как часть пакета. Это позволяет IDE подсказывать структуру, сохраняя чистое разделение код/данные.

### Исходные правила vs скомпилированные правила

- **Исходные правила** (`transliteration_rules.py`) — человекочитаемое описание с макросами и скобочными группами.
- **Скомпилированные правила** (`compiled_rules.json`) — плоский список пар `(L, R)` для быстрого применения движком.

Компиляция происходит один раз и кэшируется. При изменении исходника нужно вызвать `reload_transliteration()`.

## 🧪 Тесты

Проект покрыт 81 тестом:

| Файл                       | Что проверяет                                                          |
|----------------------------|------------------------------------------------------------------------|
| `test_compiler.py`         | Парсер правил, макросы, скобочные группы, заглавные буквы              |
| `test_transducer.py`       | Детектор regex, применение правил, NFC-нормализация                    |
| `test_eng_to_ipa_hybrid.py`| Токенизация, нормализация IPA, гибридная транскрипция                  |
| `test_transcribe.py`       | Интеграционные тесты класса `Transcriber`                             |

Запуск:
```bash
pip install -e ".[dev]"
pytest tests/ -v
```
